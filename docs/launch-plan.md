# 订餐网页上线方案

## 目标

用尽量低的成本，把订餐流程做成可上线的网站，覆盖这几件事：

- 用户下单
- 显示静态收款码
- 上传付款截图
- 管理员查看订单和付款记录
- 自动统计每日销量、成本、利润

## 推荐技术组合

### 方案 A：最低成本，推荐优先采用

- 前端：Cloudflare Pages
- 数据：Supabase Postgres
- 图片：Supabase Storage
- 登录：Supabase Auth

优点：

- 不需要自建服务器
- 成本低
- 静态站点访问快
- 后续容易扩展

适合：

- 校园订餐
- 公司内部团餐
- 小团队每日收单

## 初始化步骤

### 1. 创建 Supabase 项目

- 新建一个 Supabase 项目
- 打开 SQL Editor
- 执行 [schema.sql](/C:/project/UTM&KS launch/supabase/schema.sql)

### 2. 创建管理员账号

- 在 Authentication 里创建邮箱密码账号
- 记住这个管理员邮箱

### 3. 把账号加入管理员名单

```sql
insert into public.admin_users (user_id, email)
select id, email
from auth.users
where email = '你的管理员邮箱'
on conflict (user_id) do nothing;
```

### 4. 配置前端环境变量

```env
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=your-anon-key
```

### 5. 部署前端

- Cloudflare Pages
- Vercel
- Netlify

把上面的两个 `VITE_` 环境变量填到部署平台即可。

## 数据表建议

### `menu_master`

维护完整菜单主数据：

- 名称
- 基础售价
- 成本
- 分类
- 是否默认启用

### `daily_menu`

维护当天实际可售菜单：

- 日期
- 餐点
- 今日售价
- 是否上架

### `orders`

一笔订单一条记录：

- 订单号
- 用户姓名
- 下单时间
- 支付方式
- 支付状态
- 支付截图文件路径
- 支付回调时间

### `order_items`

一笔订单可以有多个餐点：

- 订单 ID
- 餐点 ID
- 餐点名称
- 下单时售价
- 下单时成本

### `payments`

记录上传过的付款截图：

- 订单号
- 姓名
- 支付方式
- 文件路径
- 上传时间
- 状态

### `daily_stats`

按天存日报：

- 日期
- 总售出
- 总成本
- 总利润
- 已付订单数
- 备注

## 状态设计建议

如果你追求极简运营，可以直接使用：

- `未付`
- `已付`

如果你更在意风控，建议上线版改成：

- `未付`
- `待核验`
- `已付`

这样用户上传截图后不会直接进入财务统计，管理员确认后再算利润。

当前代码已经支持这两种模式，通过 `AUTO_MARK_PAID` 配置控制。

## 支付截图处理建议

### 最省事做法

- 用户上传截图
- 文件保存到 `payment-proofs` 存储桶
- 返回文件路径
- 自动更新订单状态

当前 SQL 已经自动创建 `payment-proofs` 存储桶和相应策略。

### 更稳妥做法

- 用户上传截图
- 系统只记为 `待核验`
- 管理员在后台核对后改为 `已付`

### 第 1 阶段

先把前端页面确认下来，确保业务流程和字段完整。

### 第 2 阶段

接真实数据库和文件上传。

### 第 3 阶段

补管理员登录、角色权限和生产环境域名。

### 第 4 阶段

补自动通知、订单导出、日报推送。

## 成本预估

如果订单量不大，早期通常可以控制在：

- 前端托管：免费或极低
- 数据库与存储：免费额度或低费用
- 域名：每年几十到一百多人民币

整体非常适合先做 MVP 上线。

## 当前代码已经做好的内容
